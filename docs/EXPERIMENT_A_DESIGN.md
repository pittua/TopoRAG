# 実験A設計書 — LLM直読みベースライン（事前登録）

作成日: 2026-07-05　|　状態: **凍結前ドラフト**（実行開始前に本書を確定し、以後の変更は「変更履歴」に理由付きで記録する）

## 1. 目的

提案手法（決定的知覚層＋検索＋裁定）の価値主張の核心である
「生ネットリストの直読みは (a) 近縁回路の弁別、(b) 未知回路の棄却、(c) 再現性 で信頼できない」
を、**主張ではなく実測値**にする。論文3章の表1（3条件比較）の条件①を埋める。

## 2. 評価対象と入力

- コーパス: `real_corpus.json` の48件（in-scope 28 / out-of-scope 20）。正解定義は `real_expected.yaml` に従う（strict = expect先頭1件、lenient = expectのいずれか。既存の validate_real.py と同一基準）。
- 入力: 変換済みネットリスト JSON を**そのまま**渡す（components の type / terminals / **value を含む**）。
  - 注: 提案手法の検索は部品値を使わない（α=1.0、タグ無し）ため、直読み側には**より多くの情報**が渡る。この条件で直読みが劣るなら主張はより強く、逆に direct 側有利の不公平批判は成立しない。この非対称は論文に明記する。

## 3. 条件

### P1（主条件）: 閉集合選択＋該当なし
- DB47回路の候補一覧（circuit_id と回路名）をプロンプトに提示し、**その中から1つ選ぶか「unknown（該当なし）」と答えさせる**。
- 候補の提示順は seed=42 で1回だけシャッフルした固定順（全クエリ・全モデル共通。位置バイアスの試行間変動を排除し再現性を確保）。
- 出力は JSON 固定: `{"answer_id": "<circuit_id|unknown>", "confidence": "high|medium|low", "reason": "<1文>"}`
- 位置づけ: 条件②③と同じラベル空間での比較であり、**表1の主結果はP1**。「該当なし」を明示的に与えているため、out回路に既知ラベルを付ける誤受理は選択肢不足では説明できない。

### P2（補助条件）: 自由記述
- 候補を与えず「この回路は何か。不明なら不明と答えよ」とだけ問う。
- 位置づけ: 実利用に近い条件での命名挙動の観察。試行1回のみ、考察材料として使用。
- 注意（誠実性）: out回路には発振器・測定ベンチ等が含まれ、直読みLLMが**正しく**「発振器」と同定する可能性がある。これはハルシネーションではないため、P2のout回答は「正同定 / 誤同定 / 不明」の3分類で採点する（後述 §6.2）。論文では「直読みは記述はできるが、検証可能なラベル空間と決定的棄却に載らない」という論旨で扱い、直読みを不当に貶めない。

## 4. モデルと試行

| 項目 | 設定 |
|---|---|
| モデル | Claude（`CLILLMClient(provider="claude")`）、Gemini（同 `"gemini"`）の2種。ローカル小型（qwen系）は任意の追加 |
| 生成設定 | 各クライアントの既定値（温度指定なし＝ユーザが素で使う条件） |
| 試行数 | P1: クエリごと**3試行**　/　P2: 1試行 |
| 集約 | **多数決**（3試行の最頻値）をそのモデルの回答とする。3件全て異なる場合は「不一致」とし、in/out いずれの採点でも**不正解扱い**＋件数を別掲 |

## 5. 指標（すべて Wilson 95%CI を併記）

| 指標 | 定義 | 対象 |
|---|---|---|
| Hit@1 strict | 多数決回答 = expect先頭1件 | in 28 |
| Hit@1 lenient | 多数決回答 ∈ expect | in 28 |
| 棄却成功率 | 多数決回答 = unknown | out 20 |
| 誤受理率 | 多数決回答 = いずれかのDB id | out 20 |
| 完全一致率 | 3試行がすべて同一回答だったクエリの割合 | 全48 |
| 不一致率 | 3試行がすべて異なったクエリの割合 | 全48 |

比較先（既存実測値）: 条件②（検索のみ）in28 Hit@1 strict 85.7% / lenient 89.3%、out20 TNR 0.90（θ=0.8863）、再現性は構成上100%。

## 6. 採点規則

### 6.1 P1（機械採点）
1. 応答からJSONを抽出（コードフェンス除去→最初の `{...}` をパース）。パース不能・answer_id が候補集合∪{unknown} 外 → その試行は「無効」とし、多数決では票なし扱い。
2. 上記 §5 の定義で機械的に採点。**人手判定は発生しない**。

### 6.2 P2（規則＋記録付き人手判定）
1. 回答文を凍結済みキーワード表（`baseline_name_map.yaml`）で family に写像（例:「ローパス」→filter_passive）。
2. in: family一致で lenient 相当、expect回路名との一致（表の同義語）で strict 相当。
3. out: 「正同定（回答が回路の実体と合致。real_expected.yaml の note を正解記述とする）/ 誤同定 / 不明」を判定。キーワード表で決まらない場合のみ人手判定し、**全件の判定理由をログに残す**。
4. P2 は補助条件であり表1には載せない（本文の考察で使用）。

### 6.3 凍結物（実行前に確定し、実行後は変更しない）
- プロンプト全文（P1/P2、`run_baseline_direct.py` 内に固定文字列として同梱）
- 候補一覧の順序（seed=42）
- `baseline_name_map.yaml`（P2用同義語・キーワード表）
- 本書の指標定義・集約規則

## 7. 手順

```bash
# 1) ドライラン（モック・ネットワーク不要。配線と採点の検証）
python run_baseline_direct.py --provider mock --prompt p1 --trials 3
python score_baseline_direct.py results/baseline_p1_mock.jsonl

# 2) 本番（Claude → Gemini の順。CLIログイン済み前提・APIキー不要）
python run_baseline_direct.py --provider claude --prompt p1 --trials 3
python run_baseline_direct.py --provider gemini --prompt p1 --trials 3
python run_baseline_direct.py --provider claude --prompt p2 --trials 1
python run_baseline_direct.py --provider gemini --prompt p2 --trials 1

# 3) 採点
python score_baseline_direct.py results/baseline_p1_claude.jsonl
python score_baseline_direct.py results/baseline_p1_gemini.jsonl
```

- 生応答は全件 JSONL に保存（クエリid・試行番号・タイムスタンプ・モデル・生テキスト）。**採点は保存済みログからのみ**行い、再実行での差し替えは新しいファイルとして追加する（上書き禁止）。
- 概算負荷: P1 = 48×3×2モデル = 288呼び出し、P2 = 96呼び出し。

## 8. 解釈の事前宣言（結果を見る前に書いておく）

- 予想: P1のin28では直読みも中程度の正答（教科書回路は当たる）。差が出るのは (a) 部品集合同一・配置違いの近縁ペア、(b) out20の誤受理率、(c) 不一致率。
- **予想が外れた場合も結果はそのまま報告する**。仮に直読みが全指標で同等以上なら、論文の主張軸を「同等精度を、検証可能な根拠（IR）と決定的棄却つきで達成」（説明可能性・監査可能性）に切り替える。この撤退条件も事前に宣言しておく。
