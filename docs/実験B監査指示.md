## 背景（なぜ監査が必要か）

実験Bレポートの見出しスコア（③ strict 17/28 = 60.7%、②比 −25.0pt）には、
そのまま論文に使えない疑いが2つある。**本作業の目的は数字を良くすることではなく、
①②③の数字を「単一の凍結条件で測った、算術の閉じた値」に確定させることである。**

### 疑い1: 算術が閉じていない
② strict正解24件 − §4の破壊3件 ＋ 回復2件 = 23/28 のはずが、見出しは 17/28。
6件が内訳で説明されていない。

**仮説**: θゲート適用の非対称。②の85.7%はθゲートなしの検索順位で採点されているが、
③はθ通過が裁定の前提のため、in-scope側もゲートで棄却される。②の棄却評価は
TPR 0.75（in 28件中約7件がθ未満）であり、その7件（rectifier含む）が③で自動的に
strictミス計上されたなら 24 − 6（ゲート喪失の②正解）− 3（破壊）＋ 2（回復）= 17 で一致する。

### 疑い2: ②と③でコード・DB・θの条件が揃っていない可能性
- レポート§6.4に「特徴量凍結後の wl_kernel 導入等の影響と推測」との記述がある。
  距離計算が②測定後に変わっていれば、θ=0.8863 の校正は無効（README記載の既知問題）。
- レポート§0のDBが `sample_netlists.json` となっており、②の評価
  （`validate_real.py` 既定）で使った `features_db.json` と名前が一致しない。

## 作業1: 算術監査（JSONLから事実を確定）

1. `results/experiment_b_2026-07-08.jsonl` を解析し、in-scope 28件それぞれについて
   次の表を作成せよ:
   `query_id / topo_conf / θ判定(受理・棄却) / ②top-1 / ③多数決 / strict正解 / ②strict正否 / ③strict正否 / 変化(回復・破壊・不変・ゲート喪失)`
2. 集計して次を確定せよ:
   - in側のθ棄却件数と該当クエリ（仮説では7件）
   - θ棄却されたクエリのうち②がstrict正解だった件数（仮説では6件）
   - 17/28 = 24 − ゲート喪失 − 破壊 ＋ 回復 の等式が閉じるか
3. **同一条件比較**を算出せよ: θ通過クエリ（仮説では21件）に限定した
   ② strict / ③ strict（Wilson 95%CI付き）。これが裁定層の正味の実力である。
