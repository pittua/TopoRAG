## 3. 重要：診断の訂正（途中で1つ誤りを発見）

当初「受動0.90 vs 増幅0.67 ＝ ファミリ間スケール差」と診断し、**ファミリ相対スコア正規化**を提案した。
→ **これは誤り。撤回した。**

`circuit_rag.py:306` のスコア式 `alpha*topo + (1-alpha)*tag` と `validate_real.py`（alpha=0.9・タグ無し）から、
表示スコア = `0.9 × topo`。逆算すると：

| 回路 | 表示 | topo | DB厳密一致 |
|---|---|---|---|
| RC-Lowpass | 0.9000 | 1.000 | あり |
| NPN-Amplifier-CE | 0.8565 | 0.951 | **あり** |
| amplifier-ac | 0.671 | 0.746 | **なし**（NPN×3多段が未収録） |
| buck_conv | 0.710 | 0.789 | **なし**（簡略制御Buck） |

- **0.9000 は alpha=0.9 の天井アーティファクト**であって、ファミリ特性ではない。複数ファミリが一律 0.9 に張り付く。
- 厳密一致がある能動回路（NPN-CE/NMOS-CS）は topo 0.95 まで出る。
- **in-scope が低得点になる真因＝「DBに最近傍が無い（近傍距離が遠い）」**。ファミリのスケールではない。

### この訂正の含意

1. **ファミリ相対正規化を実装しても直らない**（前提が崩れた）。
2. **低得点 in-scope に対応するトポロジをDBに足せば top1 が上がり、分離余裕が改善する** → 当初の「DB拡張」案は in-scope の near-miss に対しては**部分的に正しかった**。
3. ただし out-scope の誤受理（`generic_opamp_bip` 等）は構造的に紛らわしく、DB拡張・閾値・特徴では**クリーンに切れない**。これは「何を in-scope と定義するか」という**スコープ定義の問題**。

### さらに本質的な制約：そもそも測れていない

- out-of-scope が **5件だけ** → TNR の95%CI [0.38, 0.96]、AUC CI [0.36, 0.88]：**無情報**。
- 分離余裕 −0.18 は in側1〜2点・out側1〜2点で決まる。`generic_opamp_bip` 1点を抜けば符号が反転する程度の脆さ。
- ネストCVバイアス +0.23 は、この極小データへの**過適合の症状**。
- → **今の n=20 で識別アルゴリズムを直しても、改善か偶然か判定不能。**「測定基盤を直す」が論理的に先。

> 反省：4点のデータに「ファミリスケール」という物語を後付けし、解法（正規化）まで作ってしまった。
> 再監査で否定された。同じ轍を踏まないこと＝**ノイズに過適合した解を作らない**。

## 4. 実機コーパス拡充のやり方

### 構成
- `real_corpus.json` … 第三者が描いた回路（TopoRAG形式）。**全クエリ タグ無し**（汎化テストのため）。
- `real_expected.yaml` … `scope: in/out` + `expect: [DB_id,...]` + `note`。
- 出典：`feastorg/KiCad-Simulation-Examples` と KiCad 同梱 `demos/simulation`。

### 統計的な目標
| n（各クラス） | 比率CI幅 | 判定 |
|---|---|---|
| 5 | ±0.3〜 | 無情報（現状の out） |
| 30 | ±0.15 | 実用最低ライン |
| 50 | ±0.1 | 安心 |

当面：**out 5→最低20（できれば30）、in 15→30。**

### 1件追加する手順
```powershell
# 1. 第三者の .kicad_sch を入手（kicad_real_test/ 等。READMEで git 追跡除外済み）
python kicad_sch_to_toporag.py path\to\NewCircuit.kicad_sch new.json   # 2. 変換
# 3. new.json の中身を real_corpus.json の circuits に追記
# 4. real_expected.yaml にラベル追加（scope / expect / note）
python validate_real.py   # 5. 再評価
python reject_eval.py
```

### 設計上の落とし穴（必読）
- **独立性**：DBを見てから回路を選ばない（自己検索化を避ける）。
- **ブラインドラベリング**：**スコアを見る前に scope を決める**。「低く出たから out」は循環論法で棄却性能を偽装する。既存コーパスに後付けラベルが混じっていないかも要点検。
- **out は多様＆境界ケース優先**：最も価値が高いのは `generic_opamp_bip` のような「構造的に in と紛らわしい out」。境界を定義するのはこれら。
- **in/out バランス**：AUCは両クラス数に効く。片方だけ増やさない。
- **easy/hard を分けて集計**：厳密一致 in（opamp-inverting 等）は Hit@1 を水増しする。hard in（amplifier-ac, smps-com）と分離して報告できると良い。
